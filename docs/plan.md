## 2. Database & Drizzle Setup

### 2.1 `drizzle.config.ts`

```typescript
import type { Config } from "drizzle-kit";

export default {
  schema: "./src/db/schema.ts",
  out: "./src/db/migrations",
  driver: "pg",
  dbCredentials: { connectionString: process.env.DATABASE_URL! },
} satisfies Config;
```

### 2.2 Schema (`src/db/schema.ts`)

Adopt the schema from the PRD verbatim:

- **`projects`** — `id` (uuid, defaultRandom), `name`, `videoUrl`, `shareToken` (unique), `editorId`, `createdAt`
- **`comments`** — `id` (serial), `projectId` (FK → projects.id, onDelete cascade), `authorName`, `content`, `timestamp` (doublePrecision), `isResolved` (nullable timestamp), `parentId` (nullable FK → comments.id for one-level threading), `createdAt`

**Addition:** Add a `parentId` column for threaded replies:

```typescript
parentId: serial('parent_id').references(() => comments.id, { onDelete: 'cascade' }),
```

### 2.3 DB Connection (`src/db/index.ts`)

- Export a `db` client using `drizzle()` + a `postgres` connection.
- Use `DATABASE_URL` env var.
- In dev, consider a local Postgres instance or Neon/Tembo for serverless.

### 2.4 Commands

```bash
npx drizzle-kit push          # Push schema to dev DB
npx drizzle-kit generate       # Generate migration files
npx drizzle-kit migrate        # Apply migrations
```

## 3. Authentication (NextAuth.js)

### 3.1 Config (`src/lib/auth.ts`)

- Use `CredentialsProvider` for simplicity (or `EmailProvider` / OAuth).
- Editors only — no client auth needed.
- Store session in the `jwt` strategy (simpler for serverless).

### 3.2 Route Protection

- `src/middleware.ts` — Check for session on `/dashboard/*` and `/projects/*`; redirect to `/login` if unauthenticated.
- Public routes under `/v/[token]` are excluded from middleware.

### 3.3 API Routes

- `src/app/api/auth/[...nextauth]/route.ts` — NextAuth catch-all handler.
- `src/app/api/auth/register/route.ts` — Optional: allow editor self-registration.

## 4. Core Feature Implementation

### 4.1 Custom Video Player

**Component tree:**

```
VideoPlayerProvider (context: currentTime, isPlaying, duration, comments)
  ├── <video> element (ref for HTMLMediaElement API)
  ├── CustomTimeline
  │   ├── <input type="range"> or custom progress bar
  │   └── Marker dots (rendered absolutely based on comment timestamps)
  └── PlaybackControls
      └── Keyboard event listeners (space, j, k, l)
```

**Key behaviors:**
- Ref the `<video>` element to call `.currentTime`, `.play()`, `.pause()`.
- On "Add Comment" click → pause video + read `currentTime`.
- `useEffect` for keyboard listeners (`space` → toggle play/pause, `j` → rewind 10s, `l` → forward 10s).
- Timeline markers: map over `comments` array, compute percentage positions (`timestamp / duration * 100`), render small dots.

### 4.2 Commenting System

**Client-side flow (public `/v/[token]` page):**

1. User clicks "Add Comment" button → video pauses → timestamp captured.
2. Comment input slides in with fields: `authorName` (text), `content` (textarea).
3. On submit → `POST /api/comments` with `{ projectId, authorName, content, timestamp }`.
4. Use `useOptimistic` to immediately append the comment to the list.
5. The comment appears in the timeline as a marker dot and in the sidebar list.

**Editor-side flow (dashboard `/projects/[id]` page):**

1. Editor sees all comments sorted by timestamp.
2. Each comment has a "Resolve" button → `PATCH /api/comments/[id]/resolve`.
3. Editor can reply to a comment (one level deep) → `POST /api/comments` with `parentId`.

### 4.3 Public Link Generation

- On project creation, generate a random token: `nanoid(12)` (no special chars).
- Format: `sh72-91sa-k182` (groups of 4 chars separated by dashes for readability).
- Store as `shareToken` in the `projects` table.
- The public URL becomes: `/v/{shareToken}`.

### 4.4 API Routes

| Method | Route | Purpose |
|---|---|---|
| `GET` | `/api/projects` | List editor's projects |
| `POST` | `/api/projects` | Create project (upload video) |
| `GET` | `/api/projects/[id]` | Get project details |
| `DELETE` | `/api/projects/[id]` | Delete project |
| `POST` | `/api/upload` | Upload video file to storage |
| `GET` | `/api/projects/public/[token]` | Fetch project by shareToken (public) |
| `POST` | `/api/comments` | Create comment |
| `GET` | `/api/comments?projectId=X` | List comments for a project |
| `PATCH` | `/api/comments/[id]/resolve` | Toggle resolved status |
| `DELETE` | `/api/comments/[id]` | Delete comment |

## 5. Page-by-Page Breakdown

### 5.1 Login Page (`/login`)
- **Layout:** Centered card on a solid background.
- **Content:** Email + password form (or "Sign in with Google").
- **State:** Redirect to `/dashboard` on success.
- **Edge cases:** Already logged in → redirect to `/dashboard` (via middleware).

### 5.2 Dashboard (`/dashboard`)
- **Content:** Grid of `ProjectCard` components.
- **Each card shows:** Project name, video thumbnail (if available), comment count, unresolved count, created date.
- **Actions:** Click card → `/projects/[id]`, "New Project" button → `/projects/new`.
- **Empty state:** Illustration + "Create your first project" CTA.
- **Loading state:** Skeleton cards (3-6 placeholders).
- **Error state:** "Failed to load projects. [Retry]" button.

### 5.3 New Project (`/projects/new`)
- **Content:** Form with project name + video upload.
- **Upload mechanism:**
  1. Editor selects a file → uploads to Uploadthing (or Mux/Cloudinary) via `POST /api/upload`.
  2. Returns a URL → stored as `videoUrl`.
  3. Option to paste a URL directly instead.
- **On submit:** Creates project in DB, generates `shareToken`, shows success with the public link + copy button.
- **Loading state:** Upload progress bar + spinner.
- **Error state:** "Upload failed — [reason]. Try again."

### 5.4 Project Detail / Editor View (`/projects/[id]`)
- **Layout:** Two-panel: video player on left, comment list on right (or stacked on mobile).
- **Video panel:** Same custom player as client view, but with editor controls.
- **Comment panel:** List of comments sorted by timestamp. Each shows:
  - Author name + timestamp badge (clickable → seek video).
  - Comment text.
  - Threaded replies.
  - "Resolve" / "Unresolve" toggle button.
  - Reply input.
- **Share button:** Top bar with copy link icon.

### 5.5 Client Review Page (`/v/[token]`)
- **Layout:** Two-panel: video player occupies most of the viewport, comment sidebar on the right (collapsible on mobile).
- **Top bar:** Project name + "Add Comment" button.
- **Video:** Custom player with timeline markers.
- **Comment sidebar:**
  - Lists comments in chronological order (by timestamp).
  - Each comment shows: author name, time into video (clickable → seek), text.
  - No resolve/reply for clients.
- **New comment flow:**
  1. Click "Add Comment" or press `C` key.
  2. Video pauses → timestamp frozen.
  3. Input appears inline with name + message fields.
  4. Submit → optimistic update → comment appears.
- **Edge cases:**
  - If video hasn't loaded yet → disable commenting.
  - If user tries to comment at 0:00 without playing → allow it (maybe they want to comment on the intro).
  - Long comment text → use max-height + scroll.

## 6. Component States

Every interactive component should handle these states:

| State | Implementation |
|---|---|
| **Loading** | Skeleton / spinner / shimmer animation |
| **Empty** | Illustration + message + action CTA |
| **Error** | Red border / toast / inline error message + retry |
| **Success / Ideal** | Normal rendering |
| **Edge cases** | Very long text (truncate + "Read more"), network failure (toast), rapid double-submit (disable button) |

### Specific components and their states:

**`VideoPlayer.tsx`**
- **Loading:** `poster` image or skeleton overlay until `canplay` event fires.
- **Error:** "Video failed to load. Check the URL or re-upload." + retry button.
- **Empty:** Not applicable (always has a URL).
- **Edge case:** Video doesn't exist (404) → show error state.

**`CommentInput.tsx`**
- **Loading:** Submit button shows spinner.
- **Empty:** Default prompt placeholder text.
- **Error:** Inline red text below input ("Comment too long" / "Name required").
- **Edge case:** Double-submit → disable button after first click.

**`CommentList.tsx`**
- **Loading:** Skeleton list items.
- **Empty:** "No comments yet. Click 'Add Comment' while watching to leave feedback."
- **Error:** "Failed to load comments. [Retry]".
- **Edge case:** 100+ comments → virtualized list or pagination (defer unless needed).

**`ProjectCard.tsx`**
- **Loading:** Skeleton card.
- **Empty:** Not applicable (handled by parent).
- **Error:** Not applicable (handled by parent).
- **Edge case:** Very long project name → truncate with ellipsis.

## 7. Data Flow & State Management

### 7.1 Server Actions (Next.js App Router)

Prefer **Server Actions** for mutations (simpler than writing API routes by hand in many cases):

```typescript
// Example: Server Action for creating a comment
"use server";
export async function createComment(formData: FormData) {
  const { projectId, authorName, content, timestamp } = parse(formData);
  // validate with Zod
  // insert into DB
  // revalidate path
}
```

### 7.2 Optimistic Updates (`useOptimistic`)

```typescript
const [optimisticComments, addOptimisticComment] = useOptimistic(
  comments,
  (state, newComment: Comment) => [...state, newComment]
);
```

- Used in both the client review page and the editor review page.
- The server action runs in the background; UI updates instantly.
- On error, roll back the optimistic comment (show a toast).

### 7.3 Data Fetching

- **Client review page:** Fetch project + comments via server component (RSC) or `fetch` on the client.
- **Dashboard:** Server component that queries DB and renders the list (fast, no client JS needed for the initial load).
- **Revalidation:** Use `revalidatePath()` or `revalidateTag()` after mutations.

## 8. File Upload Strategy

### Option A: Uploadthing (Recommended for MVP)
- Free tier: 2GB total, 1,000 files.
- Drop-in React component (`<UploadButton />`).
- Returns a URL → store as `videoUrl`.
- Handles video optimization automatically.

### Option B: Mux Video
- Purpose-built for video.
- Provides a React `<Video>` component with analytics.
- Requires credit card for free tier.

### Option C: Direct upload to S3 / Cloudinary
- More control, more setup.

**Recommendation:** Start with **Uploadthing** for speed. Can swap later.

## 12. Key Architectural Decisions

1. **Server-first rendering** — Use React Server Components for data fetching wherever possible. Client components only for interactivity.
2. **Server Actions over API routes** — Simpler for mutations within the same domain. Use API routes only for external-facing endpoints (e.g., public comment submission).
3. **`useOptimistic` over React Query** — Keep dependencies minimal. `useOptimistic` is built into React 19 / Canary. For complex caching needs, add React Query later.
4. **Credentials auth for MVP** — Simplest to set up. Add OAuth (Google/GitHub) as a later enhancement.
5. **`nanoid` for share tokens** — Compact, URL-safe, no sequential guessing.
6. **`parentId` in comments table** — Simple one-level threading without introducing a separate `replies` table.
